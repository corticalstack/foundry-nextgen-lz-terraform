## 2. Private Connectivity

> Private endpoints give each PaaS service a NIC with a private IP inside `snet-pe`. Private DNS zones ensure that the service's public FQDN resolves to that private IP inside the VNet, so no traffic leaves Azure's backbone.

### How PaaS private networking works

Azure PaaS services (`aif-core`, `aif-research`, APIM, Storage, Cosmos, Search) always exist in Microsoft's managed infrastructure, not inside your VNet. They are never "deployed into" a VNet the way a VM is. Private endpoints are NICs with private IPs that live inside your VNet (`snet-pe`), and create a private link tunnel into those managed services via Microsoft's backbone network.

```
Your VNet (snet-pe)          Microsoft backbone
┌─────────────────┐          ┌──────────────────┐
│ pe-aif-core     │──────────│ aif-core (PaaS)  │
│ (private NIC,   │ private  │ publicNetworkAccess│
│  10.0.0.x)      │  link    │  = Disabled       │
└─────────────────┘          └──────────────────┘
```

With `publicNetworkAccess: Disabled` set on the account, the public endpoint is closed — the private endpoint is the only network path to the service. So while the resource technically lives outside the VNet, it's effectively private — nothing on the public internet can reach it, and clients inside the VNet resolve its FQDN to the private endpoint IP via the DNS zones.

The architecture diagram correctly shows `aif-core` (and other PaaS services) outside the VNet box — they are PaaS resources, not VNet-injected resources.

## 6. Agent Backing Services

> These services provide durable storage for the AI Foundry Agent Service (Standard Setup). They are all fully private (public access disabled) and accessed exclusively via private endpoints.

### Why these services exist

The three resources are required because the Foundry Agent Service capability host needs durable, customer-owned storage for three categories of agent state. This is called **standard agent setup**.

| Resource | Role | What it stores |
|---|---|---|
| CosmosDB | Thread store | Conversation history, system messages, agent metadata (instructions, tool definitions) — in a database called `enterprise_memory` |
| Storage | File store | Files uploaded by developers (knowledge sources) and end-users during chat sessions |
| AI Search | Vector store | Chunked, embedded representations of files for semantic/RAG retrieval via the File Search tool |

Reference: [Standard agent setup](https://learn.microsoft.com/en-us/azure/foundry/agents/concepts/standard-agent-setup)

**Is private networking the trigger?**

Not exactly. These resources are required for standard setup (BYO, customer-owned). The alternative is **basic setup** where Microsoft manages shared, multitenant storage — and you provision nothing.

Private networking forces standard setup because once `publicNetworkAccess: Disabled` is set, the Agent Service runtime can no longer reach Microsoft-managed shared backends. All three dependencies must be reachable via private endpoints inside your VNet.

Reference: [Set up private networking for Foundry Agent Service](https://learn.microsoft.com/en-us/azure/foundry/agents/how-to/virtual-networks?view=foundry-classic)

**The capability host** ([docs](https://learn.microsoft.com/en-us/azure/foundry/agents/concepts/capability-hosts?view=foundry-classic)) is the resource that wires it all together — it is the configuration object telling the Agent Service which CosmosDB connection handles threads, which Storage connection handles files, and which AI Search connection handles vectors. In this codebase it is the `azapi_resource "capability_host"` in `modules/spoke-multi/main.tf`, created once per team project.

## 7. Agent Infrastructure — Capability Hosts & Connections

> Capability hosts and project connections are the "wiring" layer that tells AI Foundry Agent Service where its backing resources are and how to authenticate to them.

### Capability host scoping

Capability hosts are **project-scoped** resources (`Microsoft.CognitiveServices/accounts/projects/capabilityHosts`). There is one per Foundry project — not one per account. Each team project under `aif-spk` gets its own capability host (`caphostproj-{team}`), and the admin project under `aif-core` gets its own (`caphost-admin`).

The backing services (Storage, CosmosDB, AI Search) are **shared at the spoke level** — a single instance of each is provisioned for the entire spoke, used by all team project capability hosts. Isolation between teams is enforced at the data layer:

- **Storage**: ABAC condition on `Storage Blob Data Owner` restricts each project MI to containers prefixed with `{project-guid}-azureml-agent`

> **What is ABAC?** ABAC (Attribute-Based Access Control) lets you attach conditions to Azure role assignments that are evaluated against resource attributes at request time. Rather than granting `Storage Blob Data Owner` with blanket access to the whole account, a condition narrows it — e.g. "only blobs in containers whose name starts with `{project-guid}`". Two project identities can hold the same role on the same storage account, but each condition restricts them to their own container prefix. ABAC controls *on what resources* a role applies (by attribute), while RBAC controls *what actions* are allowed — Azure evaluates both together. See [Azure ABAC conditions](https://learn.microsoft.com/en-us/azure/storage/blobs/storage-auth-abac).
- **CosmosDB**: Cosmos DB SQL role assignment scoped to the `enterprise_memory` database, one per project MI — teams share the database but each MI can only access its own containers within it
- **AI Search**: `Search Index Data Contributor` and `Search Service Contributor` granted per project MI on the shared service
